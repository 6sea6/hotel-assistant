## 公交时间查询Bug修复（amap.js）

### Bug 1：isSubwayLineName 误判公交为地铁
- **根因**：`isSubwayLineName()` 用 `/地铁|号线/` 正则匹配线路名，但高德API返回的公交线路名格式是 `301路(电子科大清水河校区南二门站--地铁红高路站)`，名字中包含地铁站名"地铁红高路站"，导致正则误匹配
- **影响**：纯公交路线被误标为 `hasSubway=true`，排序时被优先选择（劣质路线排到了前面）
- **修复**：
  1. `isSubwayLineName(lineName, lineType)` 新增 `lineType` 参数，优先用高德API的 `type` 字段判断（"普通公交线路" vs "地铁线路"）
  2. 回退正则改为 `^地铁|^轨道交通|^\d+号线`（只匹配开头，避免站名误判）
  3. `extractSubwayLineNames()` 和 `extractNearestSubwayDistanceKm()` 同步传入 `busLine.type`

### Bug 2：排序策略不合理
- **根因**：路线排序严格按时间先后，1分钟差异就选更贵更远的路线。例如59分钟/4元/3段排在60分钟/2元/2段前面
- **修复**：时间差距 ≤5 分钟视为等效，然后优先选便宜、少换乘、距离短的
- 新排序规则：
  1. 有地铁优先
  2. 时间差距 >5分钟时，短时间优先
  3. 时间差距 ≤5分钟时，便宜优先
  4. 费用相同，少换乘优先
  5. 都相同，距离短优先

### 实际效果对比（成都新希望高新中心假日酒店→电子科技大学清水河校区）
| 项目 | 修复前 | 修复后 |
|------|--------|--------|
| 时间 | 59分钟 | 60分钟 |
| 费用 | 4元 | 2元 |
| 距离 | 8.963km | 7.958km |
| 换乘 | 3段 | 2段 |
| hasSubway | true(误判) | false(正确) |

## 近距离酒店步行路线（amap.js）
- **用户要求**：距离1km以内的酒店公交路线给步行
- **实现**：`getTransitInfo()` 中用 Haversine 公式计算酒店到目的地的直线距离
- 直线距离 ≤1.5km → 调用高德步行导航 API（`/v3/direction/walking`）
- 直线距离 >1.5km → 调用高德公交路线 API（原逻辑）
- 阈值 1.5km 而非 1km 的原因：维利亚酒店直线 1.325km 但公交绕路 6.4km，1km 阈值无法覆盖
- **新增函数**：
  - `haversineDistanceKm(loc1, loc2)` — 计算两经纬度坐标间直线距离
  - `fetchWalkingRoute(origin, destination, key)` — 调用高德步行 API，返回统一格式
- **hotel-data.json 更新**：维利亚酒店6条记录
  - distance: 6.4km → 1.3km
  - transport_time: 54分钟 → 18分钟
  - bus_route: 公交路线 → 步行1325米
