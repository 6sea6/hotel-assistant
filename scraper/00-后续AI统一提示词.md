# 后续 AI 统一提示词

本文件是当前采集器的交接总入口。后续 AI 接手采集器相关任务时，先读本文件，再执行任务。不要把零散日志当作业务规则源。

## 目标

你的工作只有三类：

1. 采集携程酒店数据。
2. 过滤合规房型并写入宾馆比较助手数据。
3. 维护采集器规则、比较助手字段结构与内置 AI 兜底提示词的一致性。

目标不是“尽量输出点什么”，而是稳定得到可写入、可比较、可排序的数据，并且不破坏现有结构。

## 规则优先级

按下面顺序理解规则；若其他文件与本文件冲突，以本文件为准：

1. 本文件：当前工作区唯一权威交接提示词。
2. `output/latest-run.json`：本次采集是否成功的唯一结果凭证，优先级高于终端输出。
3. 当前比较助手数据目录中的 `hotel-data.json`：比较助手当前真实数据与模板来源。
4. `src/main/services/ai-service.js`：应用内 AI 兜底实际发给模型的提示词来源。
5. `README.md`：采集器运行和排查手册，只负责补充执行细节，不再单独定义业务规则。

## 绝对禁止

1. 不要伪造价格。拿不到真实价格时，`total_price`、`daily_price` 可以为 `null`。
2. 不要伪造系统字段。不要手写 `id`、`created_at`、`updated_at`。
3. 不要绕开现有数据结构另起一套文件格式。
4. 不要把调试噪音、地铁站名、原始房型名、付款担保信息塞进 `notes`。
5. 不要只看终端输出判断成功，必须读取 `output/latest-run.json`。
6. 不要因为采集器给了 `eligible_rooms` 就跳过最终复核；采集器已做第一层规则过滤，但床型、退改与窗户信息仍要按本文件确认。
7. 不要要求人工修正同批次记录重复 id；当前写入链路会生成稳定 ID。
8. 不要手工直接改 `hotel-data.json` 里磁盘落盘的 `shared + rooms` 分组结构来回写采集结果；这很容易只改到 `shared.name`，把上一轮 `rooms` 原样带过去。

## 弱模型执行原则

假设执行任务的 AI 只有较弱的流程理解能力时，按下面的机械规则执行：

1. 缺任何前置条件就停止并汇报，不要猜。前置条件包括：模板名、有效携程详情页或列表页链接、模板匹配结果、`output/latest-run.json`、`output/<酒店名>.json` 或批量输出 JSON。
2. 采集任务只走一条固定路径：读模板 -> 执行推荐命令 -> 读 `latest-run.json` -> 读输出 JSON -> 过滤 -> 写入。不要自行发明旁路流程。
3. 非“修改规则”场景，应用内任务优先调用内置采集工具，手动排查时直接按项目内置 `scraper` 目录运行。
4. 正常流程不要使用 `--write-app-data`。这是跳过最终人工复核的危险快捷方式；只有用户明确要求并显式接受风险时，才允许和 `--unsafe-allow-unreviewed-write` 一起使用。

## 环境判定

1. 当前测试/开发环境默认使用本文件所在的 `scraper` 目录；接手任务时直接读取本目录 `00-后续AI统一提示词.md`，工作目录就是当前采集器目录，不要先探测打包路径。
2. 安装版由宾馆比较助手内置采集入口定位打包的 scraper 资源。
3. 手动排查时直接按测试/开发环境处理。

## 标准执行流程

### 场景 A：用户给“模板名 + 携程链接”，要求采集并写入

1. 先读取当前比较助手数据目录中 `hotel-data.json` 的 `templates` 数组，确认模板是否存在，并拿到：
   - `destination`
   - `check_in_date`
   - `check_out_date`
   - `room_count`
2. 运行采集命令时，优先使用 Node.js 直接执行，不要把带 `&` 的 URL 直接交给 PowerShell。
3. 推荐命令：

```bash
node src/cli.js --url "携程链接" --templateName 模板名 --auto-edge --edge-user-data-dir ./state/edge-profile --edge-profile-directory Default --edge-debugging-port 9222
```

`携程链接` 可以是酒店详情页，也可以是酒店列表页。多个详情页、多个列表页或混合粘贴文本使用：

```bash
node src/cli.js --urls "多条携程详情页/列表页链接或混合文本" --templateName 模板名 --auto-edge --edge-user-data-dir ./state/edge-profile --edge-profile-directory Default --edge-debugging-port 9222
```

列表页必须先前筛再进入详情页，前筛参数至少包括：

```bash
--targetCount 5
```

携程 URL 原生前筛可以来自用户粘贴的列表页 URL，也可以由应用内“前筛设置”或 CLI 参数写入。已识别的携程前筛包括价格、星级、排序、免费取消、点评数、评分档位、住宿类型、房型、客房特色和特色主题；未识别的 `listFilters` 片段必须原样保留。CLI 调试可用 `--priceMin`、`--priceMax`、`--starLevels`、`--sortMode`、`--freeCancel`、`--reviewCountMin`、`--ctripScoreMin`、`--accommodationTypeMode`、`--accommodationTypes`、`--roomTypes`、`--roomFeatures`、`--featureThemes`。应用内列表过滤只保留目标采集数量。

列表页解析优先使用 HTML/embedded JSON；如果页面结构变化或候选不足，启用 Edge-CDP 登录态兜底取得渲染后的 HTML，并在页面上下文中复用携程自身的 `initListRequest` 调用 `fetchHotelList` 补齐懒加载候选，再复用同一解析器。携程列表页不是传统翻页，不要通过拼接 `pageIndex`、`page` 或 `start` 参数来补页，也不要在 Node 侧裸 POST `fetchHotelList`。排查“目标 N 家但实际少于 N 家”时，优先看列表展开摘要和性能日志里的 `listApiResponseCount`、`listApiPageIndexes`、`listApiError`，不要先改详情页房型筛选规则。

如果用户在应用设置里填了高德 API Key，采集助手会传给采集器；CLI 调试时可追加 `--amapKey "你的高德Key"`。未传时是否使用采集器内置默认 Key，取决于当前安装包是否包含默认 Key。

`--auto-edge` 现在默认以后台隐藏方式启动临时 Edge 会话，采集完成后自动关闭，不应再把网页弹到前台影响用户当前操作。首次使用如果还没有可复用的 Edge profile，CLI 会先自动打开一次可见 Edge 窗口让用户登录携程；登录完成后关闭该窗口，当前任务再继续后台采集。新设备或新 profile 通常仍需要至少这一次登录准备，不能完全跳过。若怀疑某家酒店只有在登录态的可见 Edge 页面里手动打开后，目标房型价格才会真正显示出来，不要只在后台连续重试；应改为先用可见窗口重新尝试一次，确认页面上目标房型价格已可见，再继续采集。可直接给采集命令追加 `--edge-headless false`，或先运行可见的 `node src/edge-session.js --login --userDataDir ./state/edge-profile --profileDirectory Default --port 9222 --url "携程链接"` 做人工确认。

4. 采集结束后，先读 `output/latest-run.json`：
   - `success=true` 且 `eligibleCount>0`：继续。
   - `success=true` 但 `eligibleCount=0`：视为失败，不写入比较助手。
   - `success=true` 但 `totalPrice=null`：只代表部分成功，说明房价仍未拿到，需要按 README 排查；不能伪造价格。
   - `success=false`：视为失败，先排查，不写入比较助手。
5. 再读 `output/<酒店名>.json` 或 `output/batch-<模板名>.json`，逐条复核 `hotels` 和 `scrape_debug.eligible_rooms`；批量输出仍通过同一个 `hotels` 数组回写。
6. 按本文件的“房型过滤规则”和“字段填写规则”过滤与修正。
7. 只把最终合规的房型通过桥接层回写，不要手工直接修改 `hotel-data.json` 的 `shared + rooms` 结构。开发环境使用：

```bash
node src/cli.js --apply-output "output/<酒店名>.json"
```

安装版通过应用内置采集入口调用同等回写流程。
8. 正常流程不要使用 `--write-app-data`；它默认禁用，仅保留给明确接受未复核直写风险的场景。
9. 回写时只判断“本次准备写入的房型”是否与已有记录重复；如果历史房型没有出现在本次输出里，也不要自动删除或清理，除非用户明确要求清理旧数据。

### 场景 B：用户只要求采集，不要求写入

1. 同样按上面的流程执行采集。
2. 最终以 `output/latest-run.json` 和 `output/<酒店名>.json` 作为交付结果。
3. 没有用户明确要求时，不自动改比较助手数据。

### 场景 C：用户要求修改采集规则或 AI 兜底规则

1. 先改本文件。
2. 再同步修改：
   - 采集器相关实现与测试
   - `src/main/services/ai-service.js` 中的 AI 兜底提示词（仅当影响 AI 诊断时）
   - `README.md`
## 模板规则

1. 模板优先于 URL 里的日期和人数信息。
2. 先读 `templates`，再决定 `template_id`、`template_info`、`destination`。
3. 只按显式 `template_id` 或 `template_name` 精确匹配现有模板，不要再根据日期、人数、目的地自动猜模板。
4. 匹配成功：
   - 复用该模板的 `id` 到 `template_id`
   - `template_info` 必须与模板当前内容一致
   - `destination` 优先采用匹配模板的 `destination`
5. 匹配失败：
   - `template_id = null`
   - `template_info = null`
   - 再根据用户信息单独填写 `destination`
   - 不要为了兜底新建模板文件
6. CLI 也不再默认套用 `bw`；如果用户没有明确给模板，就必须自己提供 `checkIn`、`checkOut`、`roomCount`，需要路线时再补 `destination`
7. 如果最终模板里的 `destination` 为空：
   - 这是合法场景，不要报模板缺字段错误
   - 不要追问用户补 destination，也不要因为 destination 为空就判定模板匹配失败
   - `destination` 允许保持空字符串
   - 只查询宾馆附近地铁站
   - 不计算 `distance`、`transport_time`、`bus_route`

## 房型过滤规则

这些规则是写入比较助手前必须执行的业务过滤，不满足就不要写入：

1. 标注“无窗”“部分有窗”“窗户位于走廊”或“窗户位于过道”的房型，直接排除。
2. 人数筛选以模板 `room_count` 为基准：
   - `room_count = 1` 时，保留 `occupancy = 1` 的房型；也允许 `occupancy = 2` 且标准化结果为“大床房”的房型。`occupancy = 2` 的双床房、家庭房、三床房等非大床房仍排除。
   - `room_count = 3` 时，默认只保留 `occupancy = 3`；如果比较助手设置里开启“3人模板时额外保留4人房”，也允许保留 `occupancy = 4` 的房型。
   - 其他场景默认只保留 `occupancy == 模板 room_count` 的房型；人数不足或未被上述规则允许的超员房型仍排除。
3. 如果房型明确写着“3人入住且不支持 2 人”，在 2 人模板下直接排除。
4. 如果整家酒店所有候选房型都明确写了“不可取消”“不支持取消”“确认后不可退改”，或只支持“订单确认后 xx 分钟/xx 天内免费取消”这类短时免费取消，整家酒店直接跳过，不写入比较助手；入住前一天免费取消、入住当天免费取消，以及没有明确写不能取消的，都按可取消处理。
5. 采集器会先排除缺价/锁价、无有效窗、人数不匹配和取消规则不合规的房型，`eligible_rooms` 是第一层自动业务过滤后的候选；仍要人工复核床型标准化、退改与窗户信息。

## 房型标准化规则

`room_type` 只能使用统一标准化结果，不要自己发明名称：

1. 大床房：一张大床。
2. 双床房：两张床且两张床大小一致。
3. 家庭房：床型不一致，例如一张双人床加一张单人床。
4. 三床房：三张床。携程标题写“三人房”的，也统一归到“三床房”，不要写“三人房”。
5. 标题含“套房”的房型也必须继续按实际床型归类；大床→大床房，双床→双床房，不能直接输出“套房”。
6. `original_room_type` 必须原封不动保留携程原始展示名。
7. 不要只凭标题猜床型；如果标准化看起来可疑，先检查采集结果里的结构化床型信息。

## room_count 规则

1. `room_count` 记录实际入住人数，不是房型最大容量。
2. 家庭房、三床房都可能住 2 人或 3 人；没有用户明确说明时，不要按最大容量自动写 3。
3. 如果 1 人模板保留了网页显示可住 2 人的大床房，该酒店记录的 `room_count` 应记 2；如果采集结果里某个保留下来的房型明确显示可住 4 人，则该房型的 `room_count` 应记 4。
4. 模板和 CLI 的采集人数当前仍只支持 1-3；如需额外保留 4 人房，只能使用 3 人模板并开启“3人模板时额外保留4人房”。
5. 本次任务的人数，以匹配模板中的 `room_count` 为主。

## 字段填写规则

### 比较助手正式字段

写入比较助手时，至少要保持以下字段完整且结构稳定：

- `name`
- `address`
- `website`
- `total_price`
- `daily_price`
- `check_in_date`
- `check_out_date`
- `days`
- `ctrip_score`
- `destination`
- `distance`
- `subway_station`
- `subway_distance`
- `transport_time`
- `bus_route`
- `room_type`
- `original_room_type`
- `room_count`
- `room_area`
- `notes`
- `is_favorite`
- `template_id`
- `template_info`
- `id`
- `created_at`
- `updated_at`

### 采集侧附加字段

采集器输出里可能出现 `cancel_policy`、`window_status`。它们可用于筛选和复核，但当前比较助手正式字段清单不以它们作为必填主字段，因此：

1. 可以把它们当作采集判断依据。
2. 不要因为输出里带了它们，就把“比较助手正式字段表”改成另一套。
3. 如果将来要把它们正式纳入比较助手 schema，必须同步更新本文件、程序代码和相关测试。

### 日期、距离、交通、价格

1. 日期统一写 `YYYY-MM-DD`。
2. `distance`、`subway_distance`、`transport_time`、`room_area` 尽量写纯数字或纯数字字符串，不混单位。
3. `transport_time` 口径固定为“宾馆 address -> 已确认 destination”的公共交通时间；如果 `destination` 为空，则留空。
4. `subway_distance > 1.5km` 时，统一按“无较近地铁站”处理：
   - `subway_station = 无`
   - `subway_distance = 0`
5. 有公交或地铁换乘方案时，把路线文本写入 `bus_route`，支持多行；如果 `destination` 为空，则 `bus_route` 留空。
6. 同一房型有多个价格时，`daily_price` 取最低有效价。
7. `displayPrice`、`totalPrice`、`payAmount` 为 0 时，视为隐藏价，不代表免费。
8. 不能把退改罚金、担保金额、设施收费误当房价。

### notes 规则

`notes` 只写其他字段无法单独承载、但用户仍然关心的信息，例如：

- 优缺点
- 额外限制
- 早餐
- 接驳说明
- 价格获取情况

`notes` 禁止重复或混入以下内容：

1. 最近地铁站名称
2. 原始房型名
3. 公交路线正文
4. 调试信息
5. 付款担保信息，例如“到店付·需担保”
6. 把 `cancel_policy`、`window_status` 当成 notes 主体去重复抄写

如果最终还是没拿到价格，可以在 `notes` 里简要说明原因；如果房价后来已由 `api-json` 或 `edge-cdp` 补回，也要按真实情况说明，不要继续写“未获取可用房价”。

## 采集与验证规则

1. 优先使用 Edge 登录态拿真实价格。
2. `state/edge-profile/` 是登录态目录，不要随意删除。
3. 不要默认建议用户长期保留一个手工启动的 9222 常驻会话；优先使用 `--auto-edge` 的后台隐藏临时会话。首次使用没有 profile 时，CLI 会先自动弹出一次可见登录准备窗口；只有用户明确要求手动排障或重新登录时，才单独使用 `edge-session.js --login`。
4. 如果酒店疑似依赖登录态解锁价格，且当前结果仍显示 `room_price_visible=false`、`totalPrice=null`、或目标房型与页面肉眼所见不一致，优先改用可见 Edge 窗口重新尝试，人工确认目标房型价格在页面上是否真实可见，再决定是否继续排查解析逻辑。
5. 若终端输出和 `output/latest-run.json` 不一致，以 `output/latest-run.json` 为准。
6. 正常流程不要使用 `--write-app-data`；若用户明确要求危险直写，也必须同时传 `--unsafe-allow-unreviewed-write`。
7. 如果用本地 HTML 快照调试，结束前最好再跑一次远程真实采集，避免 `latest-run.json` 停留在无价快照结果。

## 你需要优先读取的文件

根据任务类型，只读必要文件，不要乱读缓存：

1. 写入比较助手前：当前比较助手数据目录中的 `hotel-data.json`
2. 采集完成后：`output/latest-run.json`
3. 复核房型时：`output/<酒店名>.json`
4. 排查采集异常时：`README.md`
5. 修改 AI 兜底规则时：`src/main/services/ai-service.js`

## 最终检查清单

提交前至少确认下面这些事：

1. 模板已先读取并匹配。
2. `destination` 来自已确认模板，而不是凭空猜。
3. 只写入业务合规房型，无窗/部分有窗和不合人数的房型已排除。
4. 如果整家酒店所有候选房型都明确不可取消，或只支持订单确认后短时间内免费取消，已经整家跳过；入住前一天免费取消或未明确写不能取消的，按可取消处理。
5. `original_room_type` 保留了携程原名。
6. `room_count` 写的是实际入住人数；宾馆记录可写到 4，但模板/CLI 采集人数仍只支持到 3。
7. `notes` 没有混入地铁站名、原始房型名、调试噪音、付款担保信息。
8. 没有伪造价格，也没有伪造系统字段。
