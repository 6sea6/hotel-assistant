# 宾馆比较助手 v8.8.8

## 项目介绍

宾馆比较助手是一款基于 Electron 开发的本地桌面程序，主要用来记录、整理、比较和导出宾馆信息。

我苦于找宾馆比较宾馆的困难所以搓了一个小程序减少工作量。

程序数据默认保存在本地，支持导入、导出和备份，方便在不同版本之间迁移，也方便后续继续维护自己的宾馆数据。

## 主要功能

### 宾馆信息管理

- 添加、编辑和删除宾馆记录
- 支持卡片视图和行式视图
- 支持单项删除和批量删除
- 可以手动刷新当前窗口中的宾馆、模板和设置数据
- 支持根据宾馆名称、携程评分、收藏状态、模板、公共交通时间、最近地铁站距离进行筛选
- 支持好评优先、低价优先、高价优先、距离近优先四种本地列表排序方式

### 模板管理

- 支持创建和复用住宿模板
- 模板中可以保存入住日期、离店日期、房间数量、目的地等信息
- 模板列表中可以直接查看入住和离店时间
- 录入宾馆时可以关联已有模板，减少重复填写

### 个性化设置

- 支持切换多套主题
- 支持自定义窗口图标
- 自定义图标会复制到当前数据目录中，删除原始图片也不会影响使用
- 可以随时恢复默认窗口图标

### 数据导入与导出

- 支持导出 JSON 备份文件
- 支持从本应用导出的 JSON 文件中恢复数据
- 支持覆盖导入和追加导入
- 导入失败时会自动回滚，避免数据只写入一半
- 自定义窗口图标也会跟随备份一起导出和恢复

### 采集助手

完整版内置采集助手资源，可以辅助处理携程酒店相关信息采集。

采集助手可以配合宾馆比较助手使用，用来辅助整理酒店名称、地址、评分、房型、价格、交通等信息。它支持携程酒店详情页和酒店列表页；列表页会先合并携程原生 URL 前筛（价格、星级、排序、免费取消、点评数、评分档位、住宿类型、房型、客房特色、特色主题），再按目标数量选择候选，最后逐个进入详情页复用原采集链路。采集和写入仍然走现有比较助手桥接层，批量任务会逐家写入并汇总本次最终写入的宾馆数和房型数。

采集助手复用 Edge 登录态获取携程真实价格；任务取消时会撤销本次已经写回的数据。设置中可以填写自己的高德 API Key，用于交通、地铁距离和路线计算；留空时是否使用程序默认 Key 取决于安装包是否内置默认 Key。

## 版本说明

宾馆比较助手 v8.8.8 主要面向采集扩充场景，重点完善了携程列表页采集、携程 URL 前筛同步、批量详情页串行编排、取消回滚、采集进度展示、自定义高德 Key、AI 接口设置回填以及说明文档同步。

当前版本只保留完整版安装包，内置采集器资源，适合手动管理宾馆数据，也适合配合采集助手整理酒店信息。

v8.8.8 的采集扩充能力包括：

- 携程详情页与列表页 URL 均可输入；列表页会展开为详情页后逐家采集
- 携程列表页 URL 前筛支持价格、星级、排序、免费取消、点评数、评分档位、住宿类型、房型、客房特色和特色主题，并保留未知筛选片段
- 列表页过滤保留目标宾馆数；住宿类型等筛选改由携程 URL 原生筛选处理
- 批量采集保持旧输出字段兼容，并新增批量统计、性能信息和最终写入摘要
- 取消任务会显示为“已取消”，并回滚本次已经写回的数据
- 设置页支持自定义高德 API Key；导出数据和 AI 工具读取设置时会脱敏该 Key
- 采集助手设置页、说明书、AI 工具说明和系统提示词同步支持列表页流程
- 左侧筛选栏移除旧的自定义权重排名，改为更直观的好评优先、低价优先、高价优先、距离近优先

## 技术栈

- **Electron**：用于构建桌面应用
- **Node.js**：用于主进程、脚本和本地能力支持
- **electron-store**：用于本地数据存储

## 运行方式

首次运行前先安装依赖：

```bash
npm install
```

启动开发版：

```bash
npm start
```

## 构建方式

执行以下命令进行构建：

```bash
npm run build
```

默认构建包含内置高德 Key。需要显式选择打包版本时：

```bash
npm run build:with-key
npm run build:no-key
```

`build:no-key` 只会在临时打包资源中移除采集器默认高德 Key，不会修改源码。无 Key 安装包输出名会带 `不含高德Key` 标记。

## 打包说明

Windows 安装包脚本：

```text
build-nsis.bat
```

默认输出目录：

```text
dist
```

默认打包图标源文件：

```text
assets/app-icon.png
```

也可以使用同名的 `jpg`、`jpeg`、`bmp`、`webp` 图片作为图标源文件。

构建时会自动生成：

- `build/icon.ico`
- `build/uninstallerIcon.ico`
- 安装页侧栏图

安装包除了比较助手本体，还会打入项目内置的 `scraper` 运行资源，包括采集器源码、采集规则说明文件 `00-后续AI统一提示词.md`，以及采集器运行所需的最小生产依赖。开发说明、测试、示例、日志、输出、状态目录和调试脚本不会进入安装包。

程序首次启动时，会把采集规则说明文件部署到当前数据目录下的 `scraper-data` 中，避免依赖构建机上的用户目录。

依赖管理采用 npm workspace：

- 根包只声明主程序依赖
- `scraper/package.json` 是采集器依赖的唯一声明来源
- 采集器依赖包括 `axios`、`cheerio`、`ws`

打包或调试过程中生成的临时验证产物，建议在验证完成后及时清理，避免工作区和磁盘体积持续增长。

## 数据导入说明

数据导入主要面向本应用自己导出的 JSON 备份文件。

导出的 JSON 文件会包含：

- `appVersion`
- `schemaVersion`
- `meta`

这些信息用于记录备份来源、应用版本和数据结构，方便后续判断文件是否适合导入。

导入时支持两种方式：

### 覆盖导入

覆盖导入会替换当前的宾馆、模板和设置数据。

适合在更换设备、恢复备份，或者希望完全使用备份文件内容时使用。

### 追加导入

追加导入只会新增宾馆和模板，保留当前设置与窗口图标。

如果导入的数据中存在宾馆或模板 ID 冲突，程序会自动重新分配新的 ID，并同步修正宾馆中的模板引用，避免导入后出现模板关联错乱。

## 数据安全

导入数据时，程序会尽量保证数据写入过程安全可靠：

- 自动归一化宾馆和模板中的 ID 类型
- 自动补齐缺失的设置默认值
- 导入过程中任一步失败，都会恢复导入前的数据
- 避免出现只写入一部分数据的半完成状态
- 自定义窗口图标会以 base64 形式写入导出 JSON 的 `meta.customAppIcon`
- 导入时会把自定义窗口图标恢复到当前数据目录中的受管图标文件里

## 适用场景

这个程序适合下面这些情况：

- 需要比较多家宾馆价格、位置和房型
- 需要反复使用相同入住日期、离店日期或目的地
- 不想把宾馆信息分散记录在网页、聊天记录或表格里
- 希望本地保存自己的住宿比较数据
- 需要导出备份，方便之后恢复或迁移
- 需要配合采集助手整理携程酒店信息
